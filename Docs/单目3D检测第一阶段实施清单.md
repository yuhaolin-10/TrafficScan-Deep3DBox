# TrafficScan 单目3D检测第一阶段实施清单

## 一、文档目的

本文档基于 [`Docs/单目3D检测分阶段优化设计文档.md`](d:\college\GraduationProject\testProject\TrafficScan_lab_copy\Docs\单目3D检测分阶段优化设计文档.md) 进一步细化第一阶段工作。第一阶段的目标是：

1. 不更换现有大模型。
2. 优先修复几何链路中的明显不稳定因素。
3. 让 3D box 从“经常明显失真”提升到“整体可用、误差可控”。

本文档重点回答两个问题：

1. 第一阶段具体要改哪些模块。
2. 每一步完成后应该怎么验收。

## 二、第一阶段总体范围

第一阶段只处理现有几何恢复链路，不涉及以下内容：

1. 不引入新的大型单目 3D 检测网络。
2. 不引入车道方向先验和 ground prior 的正式融合。
3. 不训练新的 yaw 分支或关键点网络。

第一阶段只聚焦于以下几类问题：

1. 深度采样不稳。
2. 深度融合过于激进。
3. 焦距近似过粗。
4. yaw 恢复中存在强硬的启发式偏置。
5. translation 缺少重投影一致性修正。

## 四、实施前准备

在正式动第一阶段代码之前，建议先完成下面 3 项准备工作。

### 4.1 准备固定测试集

建议从现有图片中手工挑出一小批代表性样例，至少包含：

1. 近距离车辆 3 到 5 张。
2. 中距离车辆 3 到 5 张。
3. 远距离车辆 3 到 5 张。
4. 车辆位于左车道、中间车道、应急车道附近的样例。
5. 有遮挡和无遮挡各若干张。

建议数量控制在 15 到 30 张之间，便于每次改完都能快速回归。

### 4.2 建立人工验收标准

第一阶段不适合一开始就追求完整量化指标，可以先建立肉眼可判断的验收标准：

1. 3D box 是否明显比车体大一圈以上。
2. box 底面是否明显漂浮或插入路面。
3. 朝向是否明显和车流方向相反。
4. 远车是否出现夸张的大框。

### 4.3 记录 baseline 结果

在任何代码改动前，先把当前版本在测试集上的结果导出一份，作为 baseline。

建议输出：

1. 原图。
2. 当前 3D box 结果图。
3. 每辆车的 `depth_hint`、`depth_prior`、最终深度、yaw 值。

这样后续每完成一步，都能和 baseline 对比，而不是只靠印象判断。

## 五、第一阶段实施步骤

下面把第一阶段拆成 6 个实施步骤。建议严格按顺序推进，不建议一开始同时改多个点。

## 步骤 0：补充调试输出和 baseline 记录能力

### 5.0.1 目标

在真正修几何之前，先把“错误是怎么来的”可视化出来。没有这一步，后面很容易陷入“感觉改好了，但不知道到底哪里变了”的状态。

### 5.0.2 主要改动模块

1. [`src/core/vehicle_detector_3d.py`](d:\college\GraduationProject\testProject\TrafficScan_lab_copy\src\core\vehicle_detector_3d.py)
2. [`src/main_image_test.py`](d:\college\GraduationProject\testProject\TrafficScan_lab_copy\src\main_image_test.py)
3. 可选：[`src/services/renderer.py`](d:\college\GraduationProject\testProject\TrafficScan_lab_copy\src\services\renderer.py)

### 5.0.3 具体任务

1. 给每辆车的输出结果增加调试字段，例如：
   - `depth_prior`
   - `depth_hint`
   - `depth_final`
   - `yaw_raw`
   - `yaw_final`
   - `sample_count`
   - `depth_std` 或局部方差
2. 临时提供一个开关，用于导出这些字段到日志或结果图。
3. 如果方便，可以在图上标出每辆车的深度来源和 yaw 值。

### 5.0.4 验收方式

满足以下条件即可认为通过：

1. 跑一张图后，能看到每辆车的深度相关中间值。
2. 能区分“深度来自先验还是来自 depth map”。
3. 能看到当前 yaw 的数值结果。
4. 输出不影响主流程正常运行。

---

## 步骤 1：重写 depth 采样逻辑

### 5.1.1 目标

先解决“depth_hint 容易采到路面”的问题。当前 depth 采样策略是第一阶段里最值得最先下手的点。

### 5.1.2 主要改动模块

1. [`src/core/vehicle_detector_3d.py`](d:\college\GraduationProject\testProject\TrafficScan_lab_copy\src\core\vehicle_detector_3d.py)

### 5.1.3 具体任务

重点改 `_sample_depth_hint` 这部分逻辑。

建议拆成以下小任务：

1. 将“采样区域选择”与“深度统计”拆成两个小函数，避免后面逻辑越来越乱。
2. 采样区域优先使用实例分割 mask，不优先使用整个 bbox。
3. 在 mask 内部采样时，去掉最底部一小条区域，避免直接采到路面。
4. 先做有效值过滤，再做分位数过滤，最后再算中位数。
5. 输出采样点个数、有效值比例、局部波动强弱。

### 5.1.4 这一小步的验收标准

用固定测试集检查：

1. 远处车辆的 `depth_hint` 不应频繁比 `depth_prior` 小很多。
2. 同一张图中，处于更远位置的车，其 `depth_hint` 大体上应大于近车。
3. 对遮挡车辆，若采样点不足，应能正确回退，而不是给出非常离谱的深度。
4. 调试输出里能看到采样点数不足时触发 fallback。

### 5.1.5 通过标志

如果你肉眼看到“部分明显被拉近的车，depth_hint 不再异常偏小”，就说明这一步已经有价值。

---

## 步骤 2：重构深度融合逻辑

### 5.2.1 目标

在采样更稳之后，再处理“融合权重太死”的问题。

### 5.2.2 主要改动模块

1. [`src/core/vehicle_detector_3d.py`](d:\college\GraduationProject\testProject\TrafficScan_lab_copy\src\core\vehicle_detector_3d.py)

### 5.2.3 具体任务

重点改 `compute_3d_box_on_ground` 中 depth 决策部分。

建议任务拆分如下：

1. 不再只用固定的 `DEPTH_HINT_BLEND`。
2. 引入一个“depth_hint 可信度”判断逻辑，输入可包括：
   - sample_count
   - 局部方差
   - mask 覆盖率
   - bbox 尺寸
3. 若 depth_hint 可信度低，则提高 `depth_prior` 权重。
4. 若 depth_hint 与 `depth_prior` 差异过大，则进入保守融合或触发截断。
5. 给最终结果记录 `depth_source` 或 `depth_confidence`。

### 5.2.4 验收标准

1. 在远车样例中，最终 `depth_final` 不再大面积贴近异常偏小的 `depth_hint`。
2. 对采样质量高的近车，最终 `depth_final` 仍能适度利用 depth map 信息。
3. 相比 baseline，明显超大的框数量减少。
4. 不会因为深度融合变保守而让所有框都退化成几乎一样的大小。

### 5.2.5 通过标志

固定测试集里，如果“超大框”和“贴脸放大”的现象明显减少，这一步就算达标。

---

## 步骤 3：调整焦距近似与深度先验参数

### 5.3.1 目标

在深度采样和融合变稳后，进一步减少相机内参近似过粗带来的系统误差。

### 5.3.2 主要改动模块

1. [`src/core/vehicle_detector_3d.py`](d:\college\GraduationProject\testProject\TrafficScan_lab_copy\src\core\vehicle_detector_3d.py)
2. 可选：增加一个配置文件或环境变量入口

### 5.3.3 具体任务

1. 将 `FOCAL_LENGTH_SCALE` 从硬编码常量改为更容易调试的配置项。
2. 允许在固定机位下快速试不同焦距比例。
3. 检查 `depth_h` 和 `depth_w` 的组合方式是否合理，必要时调低对宽度深度的敏感性。
4. 对不同车辆类别检查尺寸先验是否过于理想化，特别是 car 默认尺寸是否偏大。

### 5.3.4 验收标准

1. 在同一批图里，不同距离车辆的框尺寸随距离变化更自然。
2. 远车不会因为焦距估得太小而被整体拉近。
3. 同类别车辆的 3D box 大小分布更稳定，不会出现明显统一偏大。

### 5.3.5 通过标志

如果你调完参数后，发现“同一场景下几乎所有框都略微更贴车身了”，说明焦距和尺寸先验方向基本调对了。

---

## 步骤 4：重构 yaw 符号和幅值逻辑

### 5.4.1 目标

解决“朝向整体偏了”的主要问题，让 yaw 至少不再出现明显系统性偏差。

### 5.4.2 主要改动模块

1. [`src/core/vehicle_detector_3d.py`](d:\college\GraduationProject\testProject\TrafficScan_lab_copy\src\core\vehicle_detector_3d.py)

### 5.4.3 具体任务

重点改 `estimate_yaw` 相关逻辑。

建议拆分为：

1. 将 `yaw_abs` 的估计与 `yaw_sign` 的判定分开。
2. 去掉“仅按图像中心左右决定朝向正负”的硬规则。
3. 对 mask 主轴、bbox 形状、vanishing-point 几何分别计算候选朝向。
4. 设计一个简单的一致性评分，选出更合理的 yaw。
5. 必要时对小目标和远目标进一步收缩 yaw 幅度，避免远车被估成强侧视。

### 5.4.4 验收标准

1. 同一车道上多辆车的朝向应整体一致，不应左一辆右一辆明显乱跳。
2. 远处车辆的 3D box 侧面不应夸张展开。
3. 横向位置不同但真实朝向类似的车辆，不应因“在图像左边或右边”而被判成明显不同方向。
4. 相比 baseline，主观观察下“整体偏航”的感觉明显减轻。

### 5.4.5 通过标志

如果你在结果图里看到大部分车辆都“顺着车道往前走”，而不是大批量朝一个奇怪方向偏，这一步就算成功。

## 步骤 5：加入 translation 的轻量重投影修正

### 5.5.1 目标

把当前“直接摆放 3D box”的方式，提升为“先给初值，再用投影误差做小范围修正”。

### 5.5.2 主要改动模块

1. [`src/core/vehicle_detector_3d.py`](d:\college\GraduationProject\testProject\TrafficScan_lab_copy\src\core\vehicle_detector_3d.py)

### 5.5.3 具体任务

建议按保守方式实现，不要一开始就做复杂优化。

可以拆成以下小任务：

1. 把“从 depth/yaw 生成 3D box”与“评估投影误差”分开封装。
2. 定义投影误差指标，例如：
   - 投影框与原 bbox 的边界误差
   - footprint 与 mask 下缘的贴合误差
   - 底边是否贴近车辆接地位置
3. 在初始 depth 附近做小范围搜索，例如对 depth 和水平偏移做少量候选尝试。
4. 选择投影误差最小的结果作为最终 translation。

### 5.5.4 验收标准

1. footprint 更贴近车辆底部区域，不再明显漂浮。
2. 框体左右宽度与实际车体更接近。
3. 相比步骤 4 之前，少数仍偏大的框会进一步收敛。
4. 不会因为搜索修正导致运行时间显著失控。

### 5.5.5 通过标志

如果你发现某些此前“明明朝向差不多，但框总是悬着或撑太开”的车，在这一步后明显贴回去了，这说明 translation 修正已经发挥作用。

### 5.6.1 目标

把前面各小步的效果统一检验，确认第一阶段确实达成目标，而不是某一步修好了另一部分又退化了。

### 5.6.2 主要改动模块

1. [`src/main_image_test.py`](d:\college\GraduationProject\testProject\TrafficScan_lab_copy\src\main_image_test.py)
2. [`src/tests/`](d:\college\GraduationProject\testProject\TrafficScan_lab_copy\src\tests)
3. 可选：增加一个简单的回归比对脚本

### 5.6.3 具体任务

1. 用固定测试集重新导出优化后的全部结果。
2. 与 baseline 逐张对比。
3. 记录每一类典型问题是否减少：
   - 大框
   - 漂浮框
   - 远车扩张
   - 朝向偏航
4. 如果方便，可以增加简单统计：
   - depth_hint 与 depth_prior 的平均差值
   - 大框样例数量
   - 明显偏航样例数量

### 5.6.4 第一阶段最终验收标准

满足以下大部分条件即可判定第一阶段完成：

1. 固定测试集中，明显“过大”的 3D box 数量明显下降。
2. 远距离车辆不再频繁被估计得过近。
3. 车辆朝向不再出现整批系统性偏转。
4. 结果在不同距离、不同车道位置下更一致。
5. 违章判定主流程不被破坏，渲染和入库仍正常。

## 六、推荐开发顺序

建议按下面顺序推进，不要跳步：

1. 先做步骤 0，建立 baseline 和调试输出。
2. 再做步骤 1，先把 depth 采样变稳。
3. 然后做步骤 2，修融合逻辑。
4. 再做步骤 3，调焦距和先验尺寸。
5. 接着做步骤 4，修 yaw。
6. 最后做步骤 5，用重投影修正 translation。
7. 完成后统一执行步骤 6 验收。

原因是：

1. 若 depth 采样本身不稳，后面的 translation 和 yaw 调优很容易被假象干扰。
2. 若没有 baseline，对比就会失去依据。
3. yaw 和 translation 是更靠后的几何修正，应建立在深度链路相对稳定之后。

## 八、总结

第一阶段的任务不是让单目 3D 检测一下子达到论文最优，而是先把当前系统中最明显、最影响实际使用的几何误差修掉。

从工程价值来看，第一阶段最重要的不是“做了多少新算法”，而是：

1. 让 depth 采样和融合不再随便把车拉近。
2. 让 yaw 估计不再被硬规则带偏。
3. 让 3D box 至少能够在视觉上大体贴合车辆和地面。

只要第一阶段做到这三点，后面的 lane/ground prior 和轻量学习模块才有一个稳定的基础可以接上。
