# 单目3D检测阶段二细化实施方案

## 1. 这次细化后的核心判断

结合当前 GUI 现象，阶段二的主要问题不是“lane prior 方向思路错了”，而是“lane prior 的提取方式太容易被破碎边界带偏”。

当前实现的问题有两个：

1. 全局箭头会被多块应急车道区域、透视差异、另一侧边界一起带偏，不适合作为决策依据。
2. 局部箭头目前是直接在车辆锚点附近取一批 polygon 点做 PCA，这种方法对遮挡、缺口、锯齿边界比较敏感。

因此，阶段二后续优化的主线应该调整为：

`lane mask / polygon -> 提取边界 -> 对边界做平滑拟合 -> 在车辆附近求局部切线 -> 用切线约束 yaw / depth`

这条路线比直接重训大模型更适合当前项目阶段。

---

## 2. 直接采纳的决策

### 2.1 删除全局箭头显示

全局箭头已经证明会误导观察，因此直接从 GUI 渲染中移除。

处理原则：

1. 不再显示全局青色箭头。
2. 保留每辆车附近的局部箭头作为调试对象。
3. 后续所有 lane prior 决策都只依赖局部边界方向，不依赖全局主方向。

### 2.2 不优先重训车道分割模型

当前阶段先不把主要精力放在重新训练 lane segmentation 模型上，原因是：

1. 现有分割在不少图片上已经能提供可用的车道区域。
2. 当前更大的误差来自“如何从 mask 中提方向”，而不是“完全分不出来”。
3. 边界平滑与拟合的收益更直接，工程成本也更低。

---

## 3. 总体实施思路

后续阶段二建议拆成 4 个连续模块。

### 模块 A：车道边界提取

目标：
从整块 lane 区域中提取真正可用于方向估计的边界，而不是继续直接使用整块 polygon 点云。

输出：

1. lane mask 的主轮廓
2. 左右两条候选边界
3. 每条边界的长度、连续性、拟合残差

核心原则：

1. 方向信息来自边界切线，不来自整块区域散点主轴。
2. 每辆车只使用离自己最近、且质量足够好的那条边界。

### 模块 B：边界平滑拟合

目标：
让方向估计不再被“坑坑洼洼”的边界局部抖动带偏。

候选方法按优先级排序：

1. 边界重采样 + 滑动平均
2. 边界重采样 + 低阶多项式拟合
3. 边界重采样 + 样条曲线拟合

建议先做第 1 种，够简单，也最容易调试。

### 模块 C：直线/曲线双模式方向提取

目标：
针对不同路段自动选择更合适的边界模型。

规则：

1. 如果边界拟合成直线后的残差足够小，则使用直线模式。
2. 如果边界明显弯曲，或直线拟合残差较大，则使用曲线模式。
3. 最终给每辆车的不是“整条边一个固定方向”，而是“该车附近那一点的局部切线方向”。

### 模块 D：把局部切线正式接入 yaw / depth

目标：
将局部边界切线从“可视化辅助”升级为正式几何约束。

接入方式：

1. 对 yaw：作为 lane base direction，视觉信息只提供 residual。
2. 对 depth：作为 consistency 约束，而不是直接回归深度值。
3. 对置信度：边界太短、太碎、太不连续时，自动降低 lane prior 权重。

---

## 4. 具体模块拆分

## 4.1 模块 A：lane mask 后处理与边界提取

### 拟改模块

1. `src/core/lane_segmenter.py`
2. `src/core/lane_geometry.py`

### 具体工作

1. 对原始 lane mask 做轻量后处理。
2. 去掉过小连通域。
3. 适度闭运算，补齐小缺口。
4. 从 mask 中提主轮廓。
5. 将轮廓按图像上下方向重采样，得到更稳定的边界点列。

### 重点说明

这里不是为了让 mask 更好看，而是为了让边界方向更稳定。

### 验收标准

1. 同一张图里，边界不再出现明显碎裂断点。
2. 车辆遮挡导致的小缺口不会让箭头方向突然翻转。
3. GUI 上的局部箭头在相邻车辆之间更连续。

---

## 4.2 模块 B：边界平滑拟合

### 拟改模块

1. `src/core/lane_geometry.py`

### 具体工作

1. 新增边界点列平滑函数。
2. 在 y 方向重采样后，对 x(y) 做平滑。
3. 输出平滑后的边界点列。
4. 允许保留原始边界和拟合边界两套结果，便于调试可视化。

### 建议先做的实现

优先用“重采样 + 滑动平均”，因为：

1. 实现简单。
2. 参数直观。
3. 容易通过 GUI 观察调参。

### 验收标准

1. 边界整体趋势保持不变。
2. 小锯齿和局部凹坑被明显抹平。
3. 不会把本来较直的边界拟合得乱飘。

---

## 4.3 模块 C：直线/曲线双模式

### 拟改模块

1. `src/core/lane_geometry.py`

### 具体工作

1. 对边界做一次直线拟合。
2. 计算直线拟合残差。
3. 如果残差低于阈值，标记为直线模式。
4. 否则标记为曲线模式，使用平滑曲线切线。
5. 为每辆车输出：
   - 使用的是哪种模式
   - 边界残差
   - 局部切线方向
   - 该方向的置信度

### 为什么不能一律用直线

1. 有些场景在透视下边界本来就有弯曲趋势。
2. 只露出很短一段边界时，直线拟合会不稳定。
3. 拿错边时，强行用直线会让错误更“稳定”。

### 验收标准

1. 直路图片中，局部方向更统一。
2. 弯曲或远距区域中，箭头不会被强行拉成一条固定直线。
3. 边界模式切换不会造成箭头大幅跳变。

---

## 4.4 模块 D：正式接入 yaw 与 depth

### 拟改模块

1. `src/core/vehicle_detector_3d.py`
2. `src/services/pipeline.py`

### 具体工作

1. 将局部切线拆成：
   - lane base yaw
   - visual residual yaw
2. 当前的 soft blend 先保留，但结构上改得更清晰。
3. 对低质量边界自动降低权重。
4. 在 depth 选择时增加 lane-consistency 打分。
5. 若某个 depth 会导致车辆与本车道方向明显不一致，则降低该候选得分。

### 重点说明

这里不建议把 lane prior 直接变成“唯一真值”。

更合适的形式是：

`final yaw = lane base yaw + visual residual`

这样 lane 保证大方向正确，视觉信息负责细节微调。

### 验收标准

1. 同车道车辆的 yaw 明显更统一。
2. 遮挡区附近的车辆不会因为 lane 边界破损而整体偏向奇怪方向。
3. depth 不会因为 lane prior 接入而出现大范围新错误。

---

## 5. 实施顺序建议

建议按下面顺序推进，不要并行乱改。

### 第一步：删全局箭头，只保留局部箭头

目标：
先把误导性可视化拿掉，避免判断时被错误全局方向干扰。

验收：

1. GUI 不再显示青色全局箭头。
2. 局部黄色箭头仍可正常显示。

### 第二步：lane mask 后处理 + 边界提取

目标：
让局部方向建立在更稳定的边界上。

验收：

1. 分割缺口减少。
2. 同一边界的箭头不再因为局部噪声快速抖动。

### 第三步：边界平滑拟合

目标：
让局部方向对遮挡、锯齿、鼓包更鲁棒。

验收：

1. 遮挡区域附近的箭头方向更平顺。
2. 相邻车辆的箭头不会忽左忽右。

### 第四步：直线/曲线双模式

目标：
让直路和弯路分别用更合适的方向模型。

验收：

1. 直路更稳定。
2. 曲线不失真。

### 第五步：用新方向模型替换现有 lane prior

目标：
正式把新的局部切线方向接入 yaw / depth。

验收：

1. 3D box 黄色方向箭头更顺路。
2. 被遮挡但仍可见局部边界的车辆，方向不再明显跑偏。
3. 远车方向比当前版本更稳定。

---

## 6. 每一步的观察重点

后续每做完一步，都建议你用 GUI 人眼看这几类图：

1. 应急车道边界完整、分割很好的图
2. 被车辆遮挡、边界缺口明显的图
3. 远车很多、透视很强的图
4. 两侧都出现应急车道区域的图

重点看 4 个现象：

1. 箭头是否连续
2. 箭头是否顺着车道
3. 同车道车辆是否更一致
4. 是否出现少量车被过度修正

---

## 7. 阶段二细化后的成功标准

如果这轮细化做得比较成功，应该出现下面这些结果：

1. 全局箭头不再干扰观察。
2. 局部箭头对遮挡和边界破损更鲁棒。
3. 同一车道上的车辆朝向更一致。
4. 3D box 的方向主要跟随道路，而不是跟随局部噪声。
5. 为后续 ground prior 接入 depth 提供更稳的方向基础。

---

## 8. 当前最推荐的下一步

下一步最推荐直接开做的是：

1. `lane_geometry.py` 中新增“边界提取 + 平滑拟合”链路。
2. GUI 中增加一个可选调试层，显示“拟合后的边界”。
3. 先只替换箭头来源，不动 depth 逻辑。
4. 让用户先做人眼验收，再决定是否把这套边界切线正式接进 depth 评分。

这一步做完之后，再进入“lane prior 深度接入 depth”的后半段，会稳很多。
