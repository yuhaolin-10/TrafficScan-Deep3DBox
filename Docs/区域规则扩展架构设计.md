# 区域规则扩展架构设计

## 1. 目标

当前项目已经具备这些基础能力：

- 固定机位图片/视频的预览与处理
- 手动画多个 ROI 区域
- Deep3DBox 车辆检测与 3D 框结果
- 视频时序处理与车流量统计
- `scene_profile` 作为场景级配置落盘载体

下一阶段希望把“区域”从单纯的几何范围，升级成“可挂业务规则的场景对象”，为后续接入这些能力做准备：

- 违停区域
- 车牌遮挡 / 车牌缺失
- 非机动车上机动车道
- 逆行、压线、禁停区停留、专用道占用等更多规则

这里先做架构规划，不在本轮写业务代码。

## 2. 核心思路

建议把后续系统拆成 3 层：

1. 几何层
   负责定义区域长什么样，只关心 polygon、line、point，不关心违法规则。

2. 语义层
   负责说明这个区域是什么，例如“违停区”“机动车道”“非机动车禁入区”“车牌检查区”。

3. 规则层
   负责把检测结果和区域语义组合起来，输出“是否违规、违规类型、证据、置信度”。

这样做的好处是：

- 一个区域可以挂多个规则
- 同一种规则可以复用到多个区域
- UI、配置、算法三边职责清晰
- 后续新增规则时，不需要重做 ROI 编辑器

## 3. 推荐数据模型

建议把当前 `scene_profile` 中的 `parking_regions`，逐步升级成更通用的 `regions` 概念。

推荐新增的区域结构：

```json
{
  "id": "region_001",
  "name": "校门口违停区A",
  "region_type": "polygon",
  "semantic_type": "parking_control_zone",
  "points": [[100, 200], [320, 210], [300, 420], [120, 410]],
  "enabled": true,
  "source": "manual",
  "tags": ["fixed_camera", "roadside"],
  "rule_bindings": [
    {
      "rule_type": "illegal_parking",
      "enabled": true,
      "params": {
        "min_stop_seconds": 10,
        "target_classes": ["car", "truck", "bus"]
      }
    },
    {
      "rule_type": "plate_missing_or_occluded",
      "enabled": true,
      "params": {
        "ocr_required": true,
        "min_plate_confidence": 0.65
      }
    }
  ]
}
```

当前版本为了兼容老数据，可以先保留：

- `parking_regions`
- `count_lines`

后续建议演进为：

- `regions`
- `count_rules`
- `global_rules`

## 4. 规则接口建议

建议所有违规规则统一实现一个接口。

推荐抽象：

```python
class RegionRule:
    rule_type: str

    def prepare(self, scene_profile, runtime_context):
        ...

    def evaluate_frame(self, frame_context):
        ...

    def finalize_track(self, track_context):
        ...
```

三个阶段的含义：

- `prepare`
  读取场景配置、模型句柄、阈值、区域绑定关系，只做一次初始化。

- `evaluate_frame`
  每帧运行一次，适合“瞬时规则”，例如非机动车进入机动车道、车辆进入禁停区域。

- `finalize_track`
  面向时序轨迹的规则，适合“持续性规则”，例如违停 10 秒以上、连续 20 帧未识别到车牌。

## 5. 运行时上下文建议

后续不要让每个规则直接读原始大字典，建议统一封装上下文对象。

### SceneContext

- 相机 ID
- 场景配置
- 区域列表
- 各种模型开关
- 视频 FPS

### FrameContext

- 当前帧图像
- 帧号 / 时间戳
- 2D 检测结果
- 3D 框结果
- 跟踪结果
- 车牌识别结果
- 分割结果
- 当前命中的区域列表

### TrackContext

- track_id
- 历史中心点轨迹
- 历史速度
- 历史区域进入 / 离开事件
- 历史车牌 OCR 结果
- 历史违规状态

这样每条规则只拿自己需要的标准输入，不直接依赖某个具体模型输出字段名。

## 6. 区域与规则的关系

推荐支持 3 种绑定方式：

1. 单区域绑定单规则
   例如“这个 polygon 只负责违停判断”

2. 单区域绑定多规则
   例如“同一个校门口禁停区，同时检查违停和车牌遮挡”

3. 多区域组合成一条规则
   例如“机动车道区域 + 非机动车区域 + 方向线”，共同判断非机动车上机动车道

所以建议规则引擎不要写成“一个 region 对应一个 if else”，而是写成：

- 规则读取自己关心的区域类型
- 规则自己决定如何组合区域

## 7. 针对你这几个需求的接入方式

### 7.1 违停

推荐依赖：

- vehicle detection
- object tracking
- 停留时长统计
- 停车区域 polygon

规则思路：

- 车辆中心点或 footprint 与违停区有足够重叠
- 速度持续低于阈值
- 持续时长超过阈值
- 输出违停事件

### 7.2 车牌遮挡 / 缺失

推荐依赖：

- vehicle detection
- plate detector / OCR
- 车头朝向或车尾朝向判断
- 可选的“车牌检查区”

规则思路：

- 对进入某类区域的车辆执行车牌检查
- 多帧内始终未检测到可靠车牌，判定缺失或严重遮挡
- 允许用时序聚合降低单帧误判

### 7.3 非机动车上机动车道

推荐依赖：

- class detector
- 跟踪
- 机动车道 polygon
- 非机动车允许区域 polygon

规则思路：

- 非机动车目标连续多帧进入机动车道区域
- 且不在允许区域中
- 输出“非机动车上机动车道”

## 8. UI 建议

后续 UI 不建议只保留“Draw ROI”。

建议逐步演进成：

- 区域列表
- 当前选中区域属性面板
- 区域名称编辑
- 区域语义类型选择
- 规则绑定面板
- 规则参数面板

建议的交互流：

1. 在画面上新建 region
2. 在右侧列表看到 `Region 1 / Region 2 / ...`
3. 点击某个 region
4. 给它设置语义类型
5. 给它勾选一条或多条规则
6. 填规则参数
7. 保存到 `scene_profile`

## 9. 持久化建议

当前建议仍以 `scene_profile.json` 为主，因为它已经和固定机位场景强关联。

短期建议：

- 保持 JSON 文件
- 在 JSON 中增加 `regions` 和 `rule_bindings`
- 老字段继续兼容读取

中期建议：

- UI 仍编辑 JSON
- 运行时可同步写入 SQLite，方便查询和统计

长期建议：

- `scene_profile` 管场景结构
- `events` 表管实际违规结果
- `tracks` 表管时序对象

## 10. 向下兼容策略

建议按下面的顺序演进：

1. 先保留当前 `parking_regions`
2. 新代码读取时，把 `parking_regions` 自动映射为 `regions`
3. 新保存格式优先写 `regions`
4. 过渡期同时兼容老字段和新字段

这样不会把现在已经存下来的 ROI 配置一次性打坏。

## 12. 实施顺序建议

建议按下面顺序推进：

1. 先把 region 做成多区域、可选中、可删除、可保存
2. 给 region 增加稳定 ID 和名称
3. 给 region 增加语义类型
4. 抽出统一规则接口
5. 先接入违停规则
6. 再接车牌识别与车牌遮挡规则
7. 最后接非机动车上机动车道这种组合规则

## 13. 当前最推荐的落地方案

如果要兼顾“先跑起来”和“后期可扩展”，我建议：

- 现在就把 ROI 按多区域来维护
- 先把所有手动画区域统一称为 `region`
- 先不把规则写死在 UI 按钮里
- 规则统一从 `scene_profile` 的绑定配置读取

这样后面你要加新规则时，基本只需要：

- 新增一个规则类
- 给 UI 增加一个规则配置入口
- 给 `scene_profile` 增加一段绑定数据

而不用再重做底层区域系统。
