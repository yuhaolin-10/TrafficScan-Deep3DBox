# 车牌识别 PoC 调研与测试

## 结论先看

- 这条视频可以直接做车牌识别 PoC，近景和中近景车辆的牌照框基本能打到。
- 当前最适合你这个项目快速搬用的开源方案是 `HyperLPR3`，原因是它对中文车牌开箱即用，Windows/Python 试跑成本低。
- 如果后续你更看重可训练性、跨端部署和长期维护，建议考虑 `YOLO 车牌检测 + PaddleOCR/PP-OCRv5 识别` 的两阶段方案。
- 本次 5 帧测试共得到 `23` 条原始候选，合并重复框后约 `12` 个有效牌照框；其中多张近景车牌文本可读，但仍存在省份简称、个别字母和新能源位数抖动。

## 本次实验怎么做的

- 视频：`images/vedio/365A2046.MP4`
- 分辨率：`3840x2160`
- 时长：`16.2s`
- 抽帧策略：先扫 `12` 个候选时间点，再选出车辆最近、框面积最大的 `5` 帧
- 识别链路：
  1. 使用项目现成 `src/models/yolo11l-seg.pt` 检测车辆
  2. 对每辆近景车裁剪两个区域：整车区域、车身下半部分聚焦区域
  3. 使用 `HyperLPR3` 高精度模式跑车牌检测+识别
  4. 保存原图、标注图、裁剪图和 `summary.json`

## 开源方案调研

### 1. HyperLPR3

- 适合度：最高，优先推荐
- 优点：中文车牌场景开箱即用，自带检测+识别，Python 接入最省事
- 风险：项目更偏“现成能力调用”，对你后续自定义训练和模型替换的自由度不如两阶段方案
- 适合你当前阶段：非常适合，先把功能跑起来再说

### 2. PaddleOCR / PP-OCRv5

- 适合度：高，长期方案推荐
- 优点：生态完整，模型和部署方案成熟，后续可以拆成“车牌检测 + OCR + 时序融合”
- 风险：不是纯车牌专用方案，通常需要你自己补齐车牌检测模型，工程量比 HyperLPR3 大
- 适合你当前阶段：适合第二阶段，把 PoC 跑通后再升级

### 3. fast-alpr

- 适合度：中
- 优点：模块化、ONNX 友好，服务化部署思路清晰
- 风险：默认路线更偏国际场景，中文车牌通常还得自己补数据或替换识别模型
- 适合你当前阶段：除非你后面要做独立服务，不然优先级低于前两者

### 4. OpenALPR

- 适合度：低，不推荐直接上
- 优点：老牌项目，资料多
- 风险：许可证和模型年代都不太适合你这个新项目，中文车牌也不是它的强项

## 本次 5 帧测试结果

### 选中的帧

- `32` 帧，`1.28s`
- `63` 帧，`2.52s`
- `94` 帧，`3.76s`
- `187` 帧，`7.48s`
- `217` 帧，`8.68s`

### 识别概览

- `frame_0032_001.28s`：识别效果最好，至少有 4 个车牌框比较稳定
- `frame_0063_002.52s`：白车蓝牌结果较稳，`津A2H191` 连续两次一致
- `frame_0094_003.76s`：卡车/轿车远一些，但 `浙53Y522` 重复识别较一致
- `frame_0187_007.48s`：中景效果仍可用，`京QZ853Q`、`京KW9711` 比较稳定
- `frame_0217_008.68s`：右侧粉色小车绿牌效果不错，`京AG07011` 基本可读

### 代表性结果

- `京NMH121`，置信度 `0.9943`
- `津A2H191`，置信度 `0.9857`
- `浙53Y522`，置信度 `0.6811`
- `京QZ853Q`，置信度 `0.9000`
- `京KW9711`，置信度 `0.8771`
- `京AG07011`，置信度 `0.8761`

### 误差特点

- 同一车牌在“整车裁剪”和“下半部分聚焦裁剪”之间，经常会出现 1 位字符抖动
- 远景小车更容易把省份简称或新能源额外一位识错
- 有些框能打准，但文本只到“半可读”状态，这时适合依赖多帧融合而不是单帧决策

## 和现有项目怎么接

- 第一阶段建议直接复用现在这个思路：`车辆检测 -> 轨迹跟踪 -> 针对近景车辆裁剪 -> HyperLPR3 -> 用现有 plate_recognizer.py 做多帧融合`
- 第二阶段建议把 `NullPlateRecognizer` 替换成真实实现，并把 `PlateCandidate` / `fuse_plate_candidates` 接到视频处理 worker 里
- 第三阶段如果你想要更稳，可以把车牌识别限制到“疑似违章车辆”或“近景车辆”上，减少误检和算力开销
